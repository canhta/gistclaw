# GistClaw Plan 10: Manual Smoke Tests

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Verify the fully assembled binary behaves correctly end-to-end against a real Telegram bot, real opencode, and real claude CLI.

**Architecture:** Manual checklist — no automated tests. Each item describes setup, action, and expected result.

**Tech Stack:** Running binary on a VPS or dev machine with real credentials.
**Design reference:** `docs/plans/design-v3.md` §11, §12
**Depends on:** Plans 1–9 (binary must be built with `make build`)

---

## Execution Order

This plan is a sequential manual checklist. All 21 items must be verified in order. Each
item builds on the running binary. Stop and fix any failure before continuing.

---

## Pre-Flight: Build & Configure

Before running any check, ensure:

1. Binary is built: `make build` produces `bin/gistclaw` and `bin/gistclaw-hook`
2. `gistclaw-hook` is on the PATH (or symlinked) so Claude Code can find it
3. `.env` file contains valid credentials:
   - `TELEGRAM_TOKEN` — real bot token
   - `ALLOWED_USER_IDS` — your Telegram user ID
   - `OPENCODE_DIR` — directory where opencode will run
   - `CLAUDE_DIR` — directory where claude CLI will run
   - `DAILY_LIMIT_USD` — e.g. `5.00`
   - `LLM_PROVIDER` — `openai` or `openrouter`
   - `OPENAI_API_KEY` / `OPENROUTER_API_KEY` — depending on provider
   - Optional: `BRAVE_API_KEY` or `SERPER_API_KEY` for web search
4. Start the binary: `./bin/gistclaw`
5. Confirm logs show: `gateway started`, `opencode.Service ready`, `claudecode.Service ready`

## Smoke Test Checklist

### Check 1 — Flow A: `/oc` task → streaming output → "Done"

**Setup:** Binary running. OpenCode idle. No active sessions.

**Action:** Send `/oc write a hello world Go function` in Telegram.

**Expected result:**
- Within 2s: OpenCode session is created (no prior session)
- Streaming text chunks arrive as Telegram messages (split at 4096-char boundary if large)
- Final message: `✅ Done`
- Logs show `step-finish` cost tracking

---

### Check 2 — Flow A (busy): second `/oc` while first running

**Setup:** Binary running. Trigger a slow task: `/oc write 500 lines of boilerplate Go code`.

**Action:** Immediately send `/oc another task` before the first completes.

**Expected result:**
- Second message is rejected with: `⚠️ Agent is busy`
- First task continues to completion unaffected

---

### Check 3 — Flow B: permission ask → keyboard → approve → task continues

**Setup:** Binary running. OpenCode configured to ask for edit permission on new files.

**Action:** Send `/oc create a new file called test.go` (targeting a path opencode will prompt about).

**Expected result:**
- Mid-task: Telegram sends a keyboard message:
  `edit /path/to/test.go`
  Buttons: `[✅ Once]  [✅ Always]  [❌ Reject]  [⏹ Stop]`
- Tap `✅ Once`
- OpenCode receives `"reply":"once"` and continues
- Task completes with `✅ Done`
- SQLite `hitl_pending` row shows `status:resolved`

---

### Check 4 — Flow B (timeout): ask → wait → reminder → auto-reject

**Setup:** Binary running. Same setup as Check 3. Note the configured `HITL_TIMEOUT` value (default: 5 minutes) and `HITL_REMINDER_BEFORE` (default: 1 minute before timeout).

**Action:** Trigger the same permission ask. Do **not** tap any button. Wait.

**Expected result:**
- At `HITLTimeout - HITLReminderBefore`: Telegram sends `⏰ Approval pending — will auto-reject soon`
- At `HITLTimeout`: Permission is auto-rejected
- OpenCode receives a reject reply and aborts the tool call
- Telegram notifies with task failure or session idle message

---

### Check 5 — Flow C: PreToolUse hook → keyboard → deny → Claude aborts

**Setup:** Binary running. Claude Code configured with `gistclaw-hook` as PreToolUse hook.

**Action:** Send `/cc run a bash command` (or a task that triggers a shell tool call via Claude Code).

**Expected result:**
- Claude Code spawns `gistclaw-hook`; hook POSTs to `http://127.0.0.1:8765/hook/pretool`
- Telegram sends approval keyboard for the tool call
- Tap `❌ Reject`
- `gistclaw-hook` writes deny response to stdout, exits with appropriate code
- Claude Code aborts the tool call and reports the rejection
- Claude either stops or continues without that tool

---

### Check 6 — Flow D: zero-output prompt → warning

**Setup:** Binary running. OpenCode running.

**Action:** Send `/oc ` (command with a prompt that produces no output — e.g., send something that opencode processes but emits zero text parts, such as `/oc ` with a whitespace-only prompt if supported, or contrive a session that goes idle without emitting text).

> **Note:** This may require temporarily patching opencode config or using a known no-op prompt. If no reproducible no-op exists, send a minimal prompt and verify the code path manually in unit tests from Plan 6.

**Expected result:**
- Session status becomes `idle`
- No text parts were received
- Telegram message: `⚠️ Agent finished but produced no output. Check the session or resend your prompt.`
- No crash, no retry

---

### Check 7 — Flow E: web_search called → answer

**Setup:** Binary running. `BRAVE_API_KEY` or `SERPER_API_KEY` set. `web_search` tool is in registry.

**Action:** Send `what is the latest Go version?` (plain message, no command prefix).

**Expected result:**
- Typing indicator appears in Telegram
- LLM calls `web_search` with a relevant query
- Search results returned as `tool_result`
- LLM provides final text answer
- Telegram message contains a Go version answer
- Cost tracked (logged as `cost_usd`)

---

### Check 8 — Flow E (web_fetch): web_fetch called → summary

**Setup:** Same as Check 7.

**Action:** Send `summarize the content at https://go.dev/dl/` (plain message).

**Expected result:**
- LLM calls `web_fetch({url:"https://go.dev/dl/", format:"markdown"})`
- Content fetched, truncated to ≤50 KB / 2000 lines
- LLM summarizes content
- Telegram message contains a summary of go.dev/dl

---

### Check 9 — Flow E (doom-loop): 3 identical calls → loop aborted → final answer

**Setup:** Binary running with search tools enabled. This check may require a crafted prompt or a debug mode that forces tool repetition. If the LLM naturally avoids loops, use a test double or trace logs to verify the guard exists.

**Action:** Send a plain message designed to trigger repeated identical tool calls (e.g., craft a system prompt override or use a mock LLM in a local test run that always returns the same tool call).

**Expected result:**
- After 3 identical `web_search` calls with same arguments: doom-loop guard triggers
- LLM is forced to provide a final answer
- Telegram message is sent with the answer (even if imperfect)
- No infinite loop; conversation ends

---

### Check 10 — Flow F: direct answer without search

**Setup:** Binary running.

**Action:** Send `what's a good name for a Go error wrapping helper?` (plain message, no command prefix).

**Expected result:**
- LLM answers directly without calling any tool
- Telegram message contains a naming suggestion
- Cost tracked; no tool calls logged

---

### Check 11 — Flow G: multi-question → sequential → both answers received

**Setup:** Binary running. OpenCode configured with a task that triggers `question.asked` with multiple questions (requires a prompt known to produce this SSE event — e.g., `initialize a new Go project` where opencode asks for framework and test preferences).

**Action:** Send `/oc initialize a new Go project`.

**Expected result:**
- SSE fires `question.asked` with 2+ questions
- Telegram sends keyboard for question 1; user selects an answer
- Telegram sends keyboard for question 2; user selects an answer
- `hitl.Service` sends single POST `/question/<id>/reply` with both answers: `{"answers":[["choice1"],["choice2"]]}`
- Agent continues with combined answers

---

### Check 12 — Restart with pending HITL → auto-reject notification

**Setup:** Binary running. Trigger a HITL permission ask (Check 3 setup) and **do not respond** — leave keyboard open.

**Action:** Kill and restart the binary (`SIGTERM` then restart).

**Expected result:**
- On restart: `hitl.Service` calls `DrainPending()` on startup
- All pending HITL items are auto-rejected
- Operator receives Telegram notification: `⚠️ N pending HITL request(s) auto-rejected on restart`
- No stale keyboard buttons remain active (old keyboard is not re-sent)

---

### Check 13 — Cost guard: 80% → warning; 100% → warning, task continues

**Setup:** Binary running. Set `DAILY_LIMIT_USD=0.01` (very low limit) to trigger thresholds quickly.

**Action:** Send several plain chat messages or an `/oc` task that incurs cost. Watch cost accumulate.

**Expected result:**
- At 80% of `DAILY_LIMIT_USD`: Telegram message `⚠️ 80% of daily cost used ($X.XX / $Y.YY)`
- At 100%: Telegram message `⚠️ Daily cost limit reached ($X.XX / $Y.YY) — tasks still running but monitor usage`
- Task is **not** cancelled at 100% — it continues to completion
- Reset `DAILY_LIMIT_USD` to original value after this check

---

### Check 14 — SIGTERM mid-task → no dangling gistclaw-hook

**Setup:** Binary running. Start a slow `/oc` or `/cc` task.

**Action:** While the task is running, send `SIGTERM` to the binary process: `kill -TERM <pid>`.

**Expected result:**
- Binary performs graceful shutdown
- All goroutines and subprocess contexts are cancelled
- Any running `gistclaw-hook` process exits (no zombie processes)
- `ps aux | grep gistclaw-hook` returns no results after shutdown completes
- Binary exits cleanly (exit code 0 or signal-terminated, not a panic)

### Check 15 — ALLOWED_USER_IDS empty → fatal exit

**Setup:** Binary **not** running. Edit `.env` to set `ALLOWED_USER_IDS=` (empty value).

**Action:** Start the binary: `./bin/gistclaw`.

**Expected result:**
- Binary logs `FATAL: ALLOWED_USER_IDS must not be empty`
- Binary exits immediately with non-zero exit code
- No Telegram bot starts
- Restore `ALLOWED_USER_IDS` to correct value after this check

### Check 16 — Codex OAuth PKCE → token stored → plain chat works

**Setup:** Binary running with `LLM_PROVIDER=openai` and no pre-stored Codex token. Ensure `OPENAI_API_KEY` is either unset or intentionally invalid for this check.

**Action:** Send `/auth` or whichever command triggers the Codex OAuth PKCE flow (per design §8 — the flow opens a browser-like link and the user completes auth in browser, then GistClaw exchanges the code for a token).

**Expected result:**
- Telegram sends a message with the PKCE auth URL
- User visits URL, completes OAuth in browser
- GistClaw receives the callback, exchanges code for token
- Token is stored in SQLite
- Send a plain chat message; it succeeds using the stored token
- Logs show `codex_token stored`

### Check 17 — No search key → startup warning; plain chat works without web_search

**Setup:** Binary **not** running. Remove `BRAVE_API_KEY` and `SERPER_API_KEY` from `.env`.

**Action:** Start the binary: `./bin/gistclaw`.

**Expected result:**
- Startup log contains `WARN: no search API key configured — web_search tool omitted`
- Binary starts successfully (no fatal exit)
- Send a plain chat message: LLM answers without calling `web_search` (tool not in registry)
- Answer is delivered to Telegram normally
- Restore search key after this check

---

### Check 18 — hitl.Service crashes repeatedly → degraded-mode notification; gateway still running

**Setup:** Binary running. Requires inducing repeated `hitl.Service` failures. Options:
  - Temporarily patch `hitl.Service.Run` to always return an error (test build)
  - Or use a debug flag / env var that forces crashes

**Action:** Start binary with patched `hitl.Service` that crashes on every start. Wait for 10 crashes within 10s.

**Expected result:**
- `withRestart` retries with exponential backoff (initial 1s, cap 30s)
- After 10 crashes: `withRestart` returns `PermanentFailure`
- `app.Run` treats `hitl.Service` as non-critical:
  - `DrainPending()` called before notification
  - Operator receives Telegram notification: `⚠️ hitl.Service has permanently failed — running in degraded mode`
  - `gateway.Service` and other services remain running
  - Binary does NOT exit
- Restore `hitl.Service` after this check

---

### Check 19 — opencode.Service crashes repeatedly → degraded-mode notification; other services running

**Setup:** Same approach as Check 18, but patch `opencode.Service`.

**Action:** Start binary with patched `opencode.Service` that crashes 5 times within 30s.

**Expected result:**
- After 5 crashes: `withRestart` returns `PermanentFailure`
- `app.Run` treats `opencode.Service` as non-critical:
  - Operator receives Telegram notification: `⚠️ opencode.Service has permanently failed — running in degraded mode`
  - `gateway.Service`, `hitl.Service`, `claudecode.Service` remain running
  - Binary does NOT exit
  - `/oc` commands return an error message to user; other functionality still works
- Restore `opencode.Service` after this check

### Check 20 — gateway.Service permanent failure → root context cancelled → process exits → systemd restarts

**Setup:** Binary running under systemd (or simulate with a process supervisor). Patch `gateway.Service` to always return an error after N retries (unlimited retries still exhaust if all return error — or patch to return a non-transient error after 3 attempts).

**Action:** Start binary under systemd with patched `gateway.Service`. Wait for permanent failure.

**Expected result:**
- `withRestart` retries gateway (unlimited retries for normal transient failures)
- When all retries consistently fail: `PermanentFailure` is returned
- `app.Run` treats `gateway.Service` as **critical**: root context is cancelled
- All other services shut down gracefully
- Binary exits with non-zero exit code
- systemd restarts the binary (verify with `systemctl status gistclaw` showing restart count incremented)
- Restore `gateway.Service` after this check

---

### Check 21 — `agent.KindFromString` with unknown string → returns error, job not saved

**Setup:** Binary running. Access SQLite directly or use a test tool.

**Action:** Attempt to insert a scheduled job with an invalid agent kind string, e.g.:
```bash
sqlite3 ~/.local/share/gistclaw/gistclaw.db \
  "INSERT INTO jobs (id, kind, prompt, cron, enabled) VALUES ('test-1', 'badkind', 'test', '0 * * * *', 1);"
```
Then trigger the scheduler to pick up jobs, or call `agent.KindFromString("badkind")` directly via a test binary.

**Expected result:**
- `agent.KindFromString("badkind")` returns an error: `unknown agent kind: "badkind"`
- Job is not executed
- Scheduler logs `ERROR: unknown agent kind "badkind" for job test-1 — skipping`
- No panic; other jobs continue to run normally
