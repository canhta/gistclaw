# GistClaw -- Milestone 4: Stable 1.0

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Harden GistClaw for a stable 1.0 release. No new primitives. Polish onboarding, add operator tooling (doctor, backup, export), make interrupted runs and stale approvals visible and recoverable, and bring all docs into sync with the actual product.

**Architecture:** All four tasks in this milestone are pure hardening passes on existing packages. Task 12a extends the startup lifecycle and transaction commit path. Task 12b adds three new CLI subcommands that are read-only with respect to the journal. Task 12c polishes two existing web route files and their templates. Task 12d syncs prose documentation. No new packages are introduced; no runtime primitives change.

**Tech Stack:** Go 1.24+, `modernc.org/sqlite` (pure-Go SQLite, no CGO), stdlib `net/http`, Go `testing` package, `go test ./...`

**Module path:** `github.com/canhta/gistclaw`

---

## Global Guardrails (Milestone 4)

1. No new runtime primitives
2. No new connectors
3. No Phase 5 features (sharing, groups, publish-back, node-level control, compare view)
4. No new storage formats
5. No autonomous background loops
6. Doctor, backup, and export are read-only with respect to the journal — they may not modify run state
7. Export does not include raw events journal rows — only projected entities (runs, receipts, approvals)
8. Backup uses the SQLite backup API with a WAL checkpoint first, not a raw file copy while a writer is active
9. No `internal/runtime` importing `internal/web` (verify with `go list -deps ./internal/runtime/...`)
10. No journal writes outside `ConversationStore.AppendEvent`

- [ ] **Step 5: Implement backup.go**

  Create `cmd/gistclaw/backup.go` with a `runBackup` function.

  The function accepts a `--db` flag for the source database path. It opens the source database in WAL mode, issues a `PRAGMA wal_checkpoint(FULL)` to flush the WAL into the main database file, then uses the SQLite online backup API (via the `modernc.org/sqlite` driver's backup mechanism) to copy the database to a destination file. The destination filename is the source filename with a timestamp suffix appended before the `.db` extension in the format `YYYYMMDD-HHMMSS`, resulting in a name like `gistclaw.20260324-153000.db.bak`. After the copy completes, the function prints the full absolute path of the backup file to stdout and exits 0.

  Run: `go test ./cmd/gistclaw -run Backup`
  Expected: PASS for all three backup tests.

- [ ] **Step 6: Implement export.go**

  Create `cmd/gistclaw/export.go` with a `runExport` function.

  The function accepts a `--db` flag for the source database path and a required `--out` flag for the output JSON file path. If `--out` is absent, it prints a usage error to stderr and exits 1.

  The function opens the database read-only, queries the runs table, the receipts table, and the approvals table. It does not query the events table. It assembles a JSON envelope struct containing a `schema_version` string field (value `"1.0"`), a `exported_at` RFC3339 timestamp, a `runs` array, a `receipts` array, and an `approvals` array. It marshals the envelope to JSON with indentation and writes it to the output path. It prints the output path and the counts of each exported entity to stdout, then exits 0.

  Run: `go test ./cmd/gistclaw -run Export`
  Expected: PASS for all four export tests.
