Generated by /plan-design-review on 2026-03-24.
Feeds into: `docs/20-v1-implementation-backlog.md` Tasks 8, 9b, 10.

This doc is the source of truth for UI design decisions. Implement against this, not against

---

## Design Stance

This is a **local operator tool**, not a consumer product. The operator is a solo engineer
running `gistclaw serve` at localhost, usually with a terminal open beside the browser.

Design posture:
- Calm by default. The system should feel quiet when idle, urgent only when action is needed.
- Dense and informative. Power users tolerate density; whitespace-first designs hide information.
- Zero visual noise. Every UI element earns its pixels or gets cut.
- Not a marketing page. No hero sections, no 3-column feature grids, no gradients for effect.

Reference aesthetic: GitHub PRs, Grafana, self-hosted Sentry — professional operator tools,
not SaaS dashboards designed for demos.

Tech constraint: Go stdlib `html/template`, no frontend framework. Progressively enhanced
with minimal vanilla JS for SSE and form interactions. CSS only — no Tailwind, no utility
framework until it proves necessary.

## 1. Global Navigation

**Structure: Top bar, minimal**

```
┌─────────────────────────────────────────────────────────────────────────┐
│ gistclaw   Runs   Approvals ●3   Settings                        ● idle    │
├─────────────────────────────────────────────────────────────────────────┤
│ [page content]                                                           │
└─────────────────────────────────────────────────────────────────────────┘
```

Rules:
- Top bar height: 44px. Fixed position. No scrolling with content.
- Logo: "gistclaw" in monospace, 14px, subtle (not the primary focus).
- Nav items: Runs | Approvals | Settings. Plain text, no icons in v1.
- Approvals badge: orange dot + count when any approval is pending. Disappears when none.
  Badge updates in real time via SSE (same channel as run updates).
- Idle indicator (far right): small colored dot + label.
  - `● idle` — gray, when no runs are active and no model calls in flight
  - `● N active` — green, when N runs are actively running
  - `⚠ N pending` — orange, when runs are interrupted or queued (needs attention)
  Updates live via SSE.
- Active nav item: underline only (no background color change). Underline = 2px, brand color.
- Milestone 3 additions: Team | Memory added to nav between Approvals and Settings.

## 2. Runs Page

**Information hierarchy:**

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Runs                                              [+ Submit task]        │
│                                                                          │
│ ACTIVE  ──────────────────────────────────────────────────────────────  │
│  ● run-abc   repo-task   Drafting patch...        2m ago   [view]        │
│                                                                          │
│ NEEDS ATTENTION  ─────────────────────────────────────────────────────  │
│  ⚠ run-def   repo-task   Approval needed          5m ago   [view]        │
│  ⚡ run-ghi   repo-task   Interrupted              1h ago   [view]        │
│                                                                          │
│ RECENT  ──────────────────────────────────────────────────────────────  │
│  ✓ run-jkl   repo-task   Completed                3h ago   [view]        │
│  ✗ run-mno   repo-task   Failed: rate limit       1d ago   [view]        │
└─────────────────────────────────────────────────────────────────────────┘
```

Column spec (per row):
1. **Status icon + run ID** (primary): colored dot/icon + short run ID
2. **Task type**: always "repo-task" in v1 — shown for future extensibility
3. **Current status phrase** (secondary): human-readable current state or last event
4. **Age** (tertiary): relative timestamp ("2m ago", "3h ago", "1d ago")
5. **View link**: always rightmost, plain text link

Grouping (in order):
1. **ACTIVE** — running, verifying states. Updates live (row replaces in-place via SSE, no page reload).
2. **NEEDS ATTENTION** — waiting_approval, interrupted, queued. Orange/amber header.
3. **RECENT** — completed, blocked, failed. Newest first. Limit to 20 rows; "show more" link for older.

Empty state (no runs ever):
```
  You haven't run any tasks yet.
  Submit your first task below, or set up your workspace first.
  [Submit a task]  or  [Open onboarding]
```
Do not show "No items found." Do not show an empty table. Warmth + primary action required.

Empty state (runs exist but none match filter — future):
  Not needed in v1 (no filtering in v1).

**Submit task:** Clicking `[+ Submit task]` navigates to `/run` (dedicated page, not a modal or inline form).
This matches the CLI's `gistclaw run` concept and scales when the form adds fields (agent, model lane, workspace override).

Submit run page (`/run`):
```
┌─────────────────────────────────────────────────────────────────────────┐
│ ← Runs   Submit a task                                                   │
│                                                                          │
│ Task description                                                          │
│ ┌──────────────────────────────────────────────────────────────────┐    │
│ │ What would you like the team to work on?                          │    │
│ │                                                                   │    │
│ │                                                                   │    │
│ └──────────────────────────────────────────────────────────────────┘    │
│                                                                          │
│ Workspace  /Users/canh/Projects/myrepo  (from settings)                 │
│ Team       Repo Task Team  (from settings)                              │
│                                                                          │
│                           [Cancel]  [Start run →]                       │
└─────────────────────────────────────────────────────────────────────────┘
```

Rules:
- Workspace and Team are read-only on this page (change them in Settings).
- After submit: redirect immediately to the new Run detail page in ACTIVE state.
- No confirmation dialog. The run starts and the operator watches it live.
- Requires admin token (POST). If token missing/invalid: inline error above the button.

## 4. Replay Graph Visual Form

The replay graph is the signature UI element of the product. It must be specific and memorable.

**Form: Vertical flow diagram, not a node-link graph**

```
  ┌──────────────────────────────┐
  │ ◉ surface-agent              │  ← operator-facing agent (blue border)
  │   task: improve auth module  │
  └──────────────┬───────────────┘
                 │ delegated
  ┌──────────────▼───────────────┐
  │ ○ researcher      ✓ done     │  ← read-heavy agent (no border)
  │   5 tool calls · 1m 20s      │
  └──────────────┬───────────────┘
                 │ delegated
  ┌──────────────▼───────────────┐
  │ ◆ implementer   ⚠ approval   │  ← workspace-write agent (diamond border)
  │   3 tool calls · paused      │
  └──────────────┬───────────────┘
                 │ delegated
  ┌──────────────▼───────────────┐
  │ ○ verifier        ○ queued   │
  └──────────────────────────────┘
```

Rules:
- Nodes are rectangles, not circles. Enough room to show agent name + state + brief stats.
- Node border encodes capability: circle border = read-heavy, diamond accent = workspace-write,
  blue border = operator-facing.
- Node color encodes state: default = light gray bg, running = light green bg,
  needs-approval = light amber bg, completed = white with green check, failed = light red bg.
- Edges are single arrows with label ("delegated", "critiqued", "returned").
- No force-directed layout. Vertical top-down flow. Delegations go down; returns go up (dashed).
- Parallel children: side by side at the same depth level.
- Max visible depth without scroll: 4 levels. Deeper trees scroll within the graph panel.
- Clicking a node: highlights its timeline events in the timeline panel (smooth scroll, highlight).

For ACTIVE runs: nodes animate with a subtle pulsing border (CSS animation, no JS needed).
For COMPLETED runs: static, clean, no animation.

## 7. Onboarding Wizard

**Emotional arc: First impression → confidence → first moment of magic**

The wizard is 4 steps. The operator should feel: "I just set up something that knows my repo."

### Step 1: Bind workspace

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Welcome to gistclaw                                          1 of 4         │
│                                                                          │
│ First, point gistclaw at a repo.                                           │
│                                                                          │
│ Workspace path                                                           │
│ [/Users/canh/Projects/____________]  [Browse]                           │
│                                                                          │
│  ✓ git repo detected: myrepo (main branch)                              │
│  ✓ writable                                                              │
│  ✓ last commit: 2 hours ago                                             │
│                                                                          │
│                                           [Continue →]                  │
└─────────────────────────────────────────────────────────────────────────┘
```

Validation feedback is inline and immediate (on path change, not on submit).
Green checkmarks convey safety: "gistclaw knows about this repo already."

### Step 2: Repo scan (async)

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Reading your repo                                         2 of 4         │
│                                                                          │
│ ○ Scanning recent commits...    ✓ found 47 commits in last 30 days     │
│ ○ Checking open diffs...        ✓ 2 branches ahead of main             │
│ ○ Reading directory structure... ✓                                      │
│                                                                          │
│ This takes a few seconds. No model calls yet.                           │
│                                                          [automatically] │
└─────────────────────────────────────────────────────────────────────────┘
```

Rules:
- Each scan step shows in-progress → completed.
- "No model calls yet" — reassures the operator that nothing is running/costing money.
- Transitions to Step 3 automatically when scan completes.

### Step 3: Task shortlist (the hero moment of onboarding)

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Pick a first task                                         3 of 4         │
│                                                                          │
│ Based on your repo, here are three good starting points.               │
│ Your first run will be preview-only — nothing will change.             │
│                                                                          │
│ ┌──────────────────────────────────────────────────────────────────┐   │
│ │▌ EXPLAIN   Explain the auth module                               │   │
│ │            src/auth/ · 3 files · last modified 2h ago            │   │
│ └──────────────────────────────────────────────────────────────────┘   │
│ ┌──────────────────────────────────────────────────────────────────┐   │
│ │▌ REVIEW    Review the open diff on feature/add-logging           │   │
│ │            +234 lines · 2 files changed · opened yesterday        │   │
│ └──────────────────────────────────────────────────────────────────┘   │
│ ┌──────────────────────────────────────────────────────────────────┐   │
│ │▌ IMPROVE   Find the next safe improvement in src/database/       │   │
│ │            Last updated 5d ago · no recent test changes           │   │
│ └──────────────────────────────────────────────────────────────────┘   │
│                                                                          │
│                                           [← Back]  [Run preview →]    │
└─────────────────────────────────────────────────────────────────────────┘
```

Rules:
- Cards are clickable (not radio buttons). Selected card gets a blue border.
- One card must be selected to proceed.
- "preview-only" must be visible on this step — operator consents to the first run type.
- Shortlist is grounded in real repo signals (not generic). Show the evidence per card.
- No emoji anywhere in the UI. Task type is indicated by a colored 3px left border and a
  small caps label (EXPLAIN / REVIEW / IMPROVE) in text-secondary color.
- Card colors: EXPLAIN = blue left border, REVIEW = amber left border, IMPROVE = green left border.
- No emoji anywhere in the product \u2014 text-only throughout.

### Step 4: First preview run

Transitions to Run detail in ACTIVE state (3A layout). The wizard disappears.
After the run completes, the receipt is primary (3C layout). A persistent CTA appears:

```
┌──────────────────────────────────────────────────────────────────────┐
│ ✓ Your first preview is complete.                                     │
│   Want to use gistclaw from your phone? Connect Telegram DM.            │
│   [Connect Telegram →]        [Continue locally]                     │
└──────────────────────────────────────────────────────────────────────┘
```

This CTA only appears after the first successful run. Never again after dismissed.

## 8. Interaction State Coverage

| Screen / Feature         | LOADING           | EMPTY                          | ERROR                        | SUCCESS             |
|--------------------------|-------------------|--------------------------------|------------------------------|---------------------|
| Runs page                | Skeleton rows (3) | Warm empty state + CTA         | "Failed to load runs: [err]" | Live rows           |
| Run detail: active       | SSE connecting... | N/A (always a run)             | SSE lost: [reconnect btn]    | Live graph + timeline |
| Run detail: approval     | N/A               | N/A                            | Submit failed: [retry]       | ✓ inline replace    |
| Approvals page           | Skeleton rows     | "No approvals pending."        | "Failed to load: [err]"      | Inline resolution   |
| Settings: save           | [Saving...]       | N/A                            | ✗ Failed: [reason]           | ✓ Saved (fades 2s)  |
| Onboarding scan          | Step-by-step anim | N/A                            | "Scan failed: [path err]"    | Auto-advance        |
| Workspace validation     | ✓/✗ inline        | Path empty = no indicator      | ✗ Not a git repo             | ✓ green checks      |

**SSE disconnected state** (for Run detail active):
```
  ⚠ Live updates disconnected. [Reconnect]
  Last event: 14:32:30 (22s ago)
```
Auto-reconnects every 5s. Button for manual reconnect. Does NOT use an alert/modal.

**Skeleton loading:** for list pages (Runs, Approvals), show 3 skeleton rows that match
the height/layout of real rows. Not a spinner. Not "Loading...".

## 9. Why-Cards

Why-cards answer "why did this happen?" inline in the timeline. They are collapsed by default.

Format (collapsed):
```
[▶ Why did gistclaw delegate to agent-02?]
```

Format (expanded):
```
[▼ Why did gistclaw delegate to agent-02?]

  agent-01 delegated because: agent-01's tool profile is read-heavy and does
  not have workspace-write capability. The task required drafting a file change,
  which is in agent-02's declared capability. This is a configured handoff edge
  in your team spec.

  [View in team spec →]
```

Rules:
- Why-cards appear in the timeline as events (not above/beside it).
- They are driven by structured policy metadata, not generated prose.
- First why-card in every run is expanded by default (teaches the operator).
- Subsequent why-cards are collapsed by default (power user mode).
- Link to relevant team spec section when the reason references team configuration.

## 10. Team Visual Composer (Milestone 3)

Plan spec: "agent graph + handoff wiring, backed by canonical teams/ YAML"

Visual form: same node design as the replay graph, but editable.

```
  ┌──────────────────────────────┐
  │ ◉ surface-agent  [edit soul] │  ← click to open soul editor panel
  └──────────────┬───────────────┘
                 │ → agent-01 [×]         ← click [×] to remove handoff edge
  ┌──────────────▼───────────────┐
  │ ○ researcher      [edit soul]│
  └──────────────┬──┬────────────┘
         ┌───────┘  └───────┐
  ┌──────▼──────┐    ┌──────▼──────┐
  │ ◆ implement │    │ ○ reviewer  │    ← parallel children
  └─────────────┘    └─────────────┘
       [+ Add agent]
```

Rules:
- Nodes are the same visual vocabulary as the replay graph (consistency is trust).
- Click a node: opens a right panel with structured soul fields (not a modal, not a full page).
- Add handoff edge: drag from node to node (or button-based for keyboard users).
- No drag-and-drop for node position — position is computed from the tree structure.
- Save validates against team spec schema before writing to disk. Errors shown inline.
- "No unsaved changes" indicator in the top bar of the composer. "● 3 unsaved changes" if dirty.

## 12. Responsive Behavior

**Primary viewport: 1280px+ (desktop, terminal beside browser)**

The UI is designed for desktop-first. An operator running `gistclaw serve` will have
a terminal open beside the browser. No mobile optimization in v1.

Minimum supported width: **1024px**. Below 1024px: a static banner reads
"gistclaw is designed for desktop. Resize your browser or use the CLI."
Do not attempt to reflow the two-panel layouts below 1024px.

**Keyboard navigation (required in v1):**
- All nav links: keyboard-focusable, visible focus ring (2px solid, offset 2px)
- Approval actions (Approve/Deny): keyboard-accessible. `Enter` = Approve, `Shift+Enter` = Deny
  when the approval card is focused.
- Forms: standard tab order, no skip links needed for single-operator tool
- Why-cards: space bar toggles expand/collapse when focused
- Focus management: after an approval resolves, focus moves to the next pending approval (or to
  the "no pending approvals" message if the queue empties)

**ARIA landmarks (required in v1):**
- `<nav>` for the top bar
- `<main>` for page content
- `<aside>` for the timeline panel (in run detail)
- Approval cards: `role="alert"` when they appear (they are time-sensitive)
- SSE disconnected banner: `role="alert"`, `aria-live="assertive"`
- Live run status updates in timeline: `aria-live="polite"`

**Color contrast:** All text must meet WCAG AA (4.5:1 for body text, 3:1 for large text).
Status colors (green/orange/red) must not be the ONLY indicator of meaning — always pair
with an icon or text label.

## 13. Design Token Minimums (no DESIGN.md yet)

Until a full `/design-consultation` is run, implement with these minimums:

**Typography:**
- Font stack: `system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`
- Monospace stack: `"SF Mono", "JetBrains Mono", "Fira Code", Consolas, monospace`
  (used for: run IDs, code diffs, shell commands, API keys, why-card code references)
- Base size: 14px. Line height: 1.5.
- No custom web fonts in v1.

**Color (minimal palette — to be replaced by DESIGN.md):**
- Background: #fafafa (near-white, not pure white — reduces eye strain)
- Surface: #ffffff (cards, panels)
- Border: #e2e8f0 (light gray, subtle)
- Text primary: #1a202c (near-black)
- Text secondary: #718096 (muted, for timestamps, labels)
- Brand/accent: #3b82f6 (blue — active nav underline, selected card border, primary actions)
- Success: #22c55e (green — active runs, completed, approved)
- Warning: #f59e0b (amber — needs attention, pending approvals, interrupted)
- Error: #ef4444 (red — failed, denied)
- Running pulse: CSS animation `opacity 1 → 0.5 → 1`, 1.5s infinite, on active node border

**Spacing scale:** 4px base. Use multiples: 4, 8, 12, 16, 24, 32, 48px.
Do not invent arbitrary spacing. Every margin/padding is a multiple of 4.

**Border radius:** 6px for cards and panels. 4px for buttons and inputs. 0px for nav items.

**Shadows:** Avoid drop shadows except for modal-like overlays (none in v1). Use borders instead.
